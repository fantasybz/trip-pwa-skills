## Three architecture decisions — LOCKED (plan-eng-review 2026-06-06)

**D1 = A** (in-place show/hide) · **D2 = A** (flat category list) · **D3 = A**
(render-side empty affordance, honest data). Rationale per option below.

### D1 — view-switching mechanism → LOCKED: A
The bottom-nav has no behaviour. Need a way to switch 行程 ↔ 美食 (and later
prep/map) that fits vanilla/offline/a11y.

- **A) in-place section show/hide** (recommended): a `view` state var; nav click
  shows/hides the schedule cluster vs a `#food-view` container; re-render target
  on switch; move focus to the view heading; nav uses `aria-current="page"`.
  No router, no extra fetch, no new page. [Layer 1 boring]
- B) hash routing (`#/food`): deep-linkable + back-button, but adds a router +
  history handling for a need v0.2 doesn't have.
- C) separate `food.html`: simplest per-page, but full reload + re-fetch + loses
  the single-shell model; duplicates header/nav.

### D2 — unanchored food placement → LOCKED: A
food-ingest writes `day_keys:[]`, `anchor:''` (the dogfood never bound a day —
that is the whole feed_candidates rationale; users paste before knowing the day).

- **A) flat list grouped by category** (recommended): the food view is the home
  for ALL ingested food; group by `category` (restaurant/cafe/…); when an entry
  HAS `day_keys`, show a "Day N" chip. food-ingest MAY gain an optional `--day`
  (additive), but the view must handle unanchored as the default.
- B) group by day + a "未排入行程" catch-all: assumes day-binding is common; today
  nothing binds, so it would be one big catch-all + empty days.
- C) require `--day` at ingest: re-introduces the v1 wedge the project already
  rejected (forces placement before the user knows it).

### D3 — contingency empty-state → LOCKED: A
CLAUDE.md: contingency is "always-visible, never hide curatorial depth." Empty
`alternatives:[]` currently renders nothing.

- **A) render-side empty affordance** (recommended): when alternatives is empty,
  render a muted "備案待補" affordance (always-visible, signals the slot). Works
  for drafted AND hand-filled days. draft-days keeps `alternatives:[]` (honest
  data — no fake entries polluting the corpus that promote/ingest must special-case).
- B) draft-days seeds a starter `{reason:'備案待補'}` stub: a chip renders, but
  fake content masquerades as real; pollutes data; render.js:102 can't tell stub
  from real.
- C) both A and B.

## Performance review

Single-user mobile PWA, corpus is curation-capped (tens of venues). Flat food
list render is O(n) over a small n — no N+1, no DB, no memory concern.
`renderFood` runs once per view-switch (re-render on switch is fine at this size).
**No issues.**
