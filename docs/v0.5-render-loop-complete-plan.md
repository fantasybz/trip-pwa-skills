# trip-pwa-skills v0.5 — Finish the Render Loop (plan)

Continuation of the v0.2 render-loop thesis (`docs/v0.2-render-loop-plan.md`):
*ingested content must surface, or ingest out-runs render.* v0.2 closed the loop
for **food**. v0.5 closes it for the **other four venue corpora** the router
already classifies but nothing renders: **desserts / attractions / fandom /
nearby**.

The router keyword-routes **5 venue/place corpora** (`router.ts:107` PRIORITY =
attractions/desserts/fandom/food/nearby; `router.ts:109` KEYWORDS excludes
`refs`). `refs` sits in the `CorpusName` type union (`router.ts:18`) but is NOT
keyword-routed — it's filled by refs-ingest (YouTube/blog URLs) and already
renders via the 今晚先看 collapse. So 5 routable corpora, only **food** of which
homes + renders end-to-end. The rest of the pipeline — scaffold, ingest, promote,
render — is **2-way** (food.json vs feed_candidates.json). A caption the router
correctly reads as a dessert cafe,
a teamLab attraction, a Pokémon Center, or a 神社 has **no home file, no promote
path, and no view**. It lands in `feed_candidates.json`, gets swept into the
美食 view as 待分類 (`render.js:179-192`), and stays there forever — the
美食 view is a junk drawer and curation can never be completed.

This is the carried-forward half of 6-persona dogfood **finding #3**
(`docs/dogfood-results-20260606-personas.md:39` — "non-food candidates
desserts/nearby are invisible → silent data loss"). v0.3.2 mitigated it
(candidates show in the food view) but did not *close* it: there is still no
proper corpus home and no way to sort a candidate into one.

Constraints (carried from approved v0.1/v0.2 design): vanilla static, no build
chain, offline-first, a11y behavioral baseline (focus-visible / tablist arrow
nav / synthetic-click / persisted state), visual variant A fixed. **Zero change
to the live Tokyo PWA (`public/`)**; runs through bundle CI (#397).

## Architecture decisions — LOCKED (`/plan-eng-review` 2026-06-11)

Three decisions mirror v0.2's D1/D2/D3 shape. **D1=A · D2=A · D3=A · scope=all 4
corpora** (locked by user). Rationale per option below.

### D1 — surface model for 5 venue corpora → LOCKED: **A** (unified view + corpus registry)
Five venue corpora (food + the 4 new) need a home. A bottom nav can't hold 7
buttons on a 375px iPhone (mobile-first primary, `CLAUDE.md` device targets).

- **A) ONE unified "口袋名單" venue view, driven by a CORPUS registry** (recommended):
  a `VENUE_CORPORA` list (`{key, file, label, icon}`) drives `loadJson` + render.
  Generalize `renderFood` → `renderVenues(sections)`; each corpus is a labeled
  section (美食 first, keeping its v0.2.3 address/hours/📍 treatment, then 甜點 /
  景點 / IP·主題 / 周邊). Bottom nav stays 3 buttons: 行程 / 地圖 / 口袋名單.
  Adding a 6th corpus later = one registry row, no nav change. Curation-uniform:
  food isn't privileged over desserts. [Layer-1 boring, DRY, mobile-clean, scales]
  - **Impl refinement (codex outside-voice):** KEEP the nav key `data-corpus="food"`,
    relabel the button text to 口袋名單. The click whitelist (`render.js:443`
    `schedule|food|map`), `setNavAvailability` key, and every v0.2 food-view test
    select on `food` — migrating the key to `venues` is needless churn that breaks
    proven tests for zero user benefit. `setView('food')` now calls `renderVenues`.
- B) Keep 美食 + add one 探索 catch-all for the other 4 → 4 nav buttons; food
  privileged arbitrarily, 探索 a vague bucket that re-creates the junk drawer.
- C) One nav button per corpus → 6-7 buttons; mobile bottom-nav bloat. Rejected.

### D2 — write path to the 4 new corpora → LOCKED: **A** (scaffold creates files + promote `--to <corpus>`; defer ingest auto-route)
The 4 files don't exist and nothing writes them. placement-promote already
gates this exact feature on "until those corpora render."

- **A) scaffold + templates/data create empty `desserts/nearby/attractions/fandom.json`;
  placement-promote gains `--to desserts|nearby|attractions|fandom`** (recommended):
  unblocks the explicitly-deferred promote target now that render exists. Loop
  closes end-to-end: ingest → 待分類 candidate → `placement-promote --to 甜點` →
  clean 甜點 section. food-ingest keeps its food+candidates behaviour (auto-routing
  to 6 corpora is additive → **v0.5.1**, mirrors how v0.2 split render (v0.2.0)
  from router-fix (v0.2.1)). SW precache is free (`regenerate-sw.ts:75`).
- B) Also generalize food-ingest to honor the router's 6-way corpus in this PR →
  bigger blast radius, mixes the render-loop change with the ingest-routing change
  v0.2 deliberately separated.
- C) Render + hand-edit only, no promote path → candidate stuck in 待分類 forever;
  curation never completes. Rejected.

### D3 — candidate (待分類) handling once corpora have homes → LOCKED: **A** (single 待分類 section, render once)
Today `collectFood` sweeps every candidate into the food view. With real corpus
sections, a candidate must not double-render.

- **A) the unified view renders feed_candidates ONCE in a trailing 待分類 section**
  (recommended): generalize `collectFood` → `collectVenues` that returns
  `{ corpusSections, candidates }`; corpus sections show only confirmed corpus
  files; candidates pool in one 待分類 block, each tagged with the router's guess
  (`candidate_for`) + a "用 placement-promote 歸位" hint. Scannable "what still
  needs sorting." No double-render.
- B) Show each candidate under its *guessed* corpus section (muted) → mixes
  unconfirmed with curated; pollutes the section; harder to see the sort backlog.
- C) Keep candidates only in the food view as today → inconsistent once the other
  corpora have homes. Rejected.

### D4 (minor) — map view → fold into D1's registry
`renderMap` already lists food + feed_candidates (incl. non-food) + anchors —
it consumes `foodItems = collectFood(food, candidates)` (`render.js:367,295`).
What it MISSES is the **confirmed corpus files** (desserts/attractions/fandom/
nearby.json) — those aren't in `foodItems`. Finishing the loop = feed the map
the same registry-loaded corpora (all mappable rows: `maps_query` || `address`),
icon-per-corpus, anchors + existing candidate handling unchanged. No new
decision — same registry.

## Code Quality review

- **DRY (the whole point of D1):** one `renderVenues` + one `VENUE_CORPORA`
  registry replaces N bespoke `renderDesserts/renderNearby/...` functions.
  MUST reuse `safeUrl` (`render.js:37`), `escapeHtml` (`:28`), `loadJson` (`:46`),
  `mapsLinkHtml`/`mapsSearchUrl` (`:261-268`), `dayLabel` (`:165`). The current
  `renderFood` item template (`:218-251`) is the generic row — extract it. (9/10)
- **Field heterogeneity, degrade gracefully:** corpora carry different fields
  (attractions: hook/area; nearby: type; desserts: area). The generic row reads a
  common subset (name_zh|name|name_jp_or_local, maps_query|address→📍,
  day_keys→chip, source_url→↗, a why/hook field, address/hours/price meta) and
  omits what's absent — exactly as `renderFood` already does defensively. (9/10)
- **One nav-state owner:** `setNavAvailability` (`:346`) + `setNavCurrent` (`:354`)
  extend to the `venues` key; do not fork. `venues` enabled iff any venue corpus
  OR a candidate has content. (8/10)
- **promote keys = registry keys, across a runtime boundary (codex #8):** the
  render registry lives in browser ESM (`render.js`, a copied template); scaffold +
  placement-promote are Bun TS. There is NO shared import between them — "single
  source of truth" is aspirational unless enforced. Decision: define the key list
  ONCE in `skills/_lib/corpora.ts` (Bun side imports it for scaffold + promote),
  `render.js` hardcodes the same 5 keys, and an integrity test asserts the three
  lists (render registry · promote `--to` enum · scaffold file set) are identical.
  Cheaper + more boring than inventing a shared JSON config format. → T-parity. (8/10)
- **per-corpus write shape (codex #6):** placement-promote currently always writes
  a food-shaped record (`category:restaurant`, `kid_friendly`, `backup_fit`,
  `last_verified` — `placement-promote.ts:89`). Widening `--to` blindly pollutes
  desserts/attractions/fandom/nearby with food fields. A non-food `--to` writes a
  generic shape (name / source_url / maps_query|address / why|hook / day_keys) and
  does NOT force the food-only fields. → T6. (9/10)

## Performance review

Single-user mobile PWA; each corpus is curation-capped (tens of venues). Render is
O(total venues) over a small n. No N+1, no DB. `renderVenues` runs once per
view-switch.

**One real issue (codex outside-voice):** `renderApp` loads JSON **sequentially**
today (`render.js:362-366` are serial `await loadJson`). Adding 4 more corpus
files makes it 9 serial fetches — on a cold/bad mobile connection each can hit the
SW `networkFirstWithTimeout` path, so the timeouts **stack** (up to ~9× worst
case). **Fix:** load the registry corpora with a single `Promise.all` (trip/days/
refs can join the same batch). O(1) round-trips instead of O(corpora). → folded
into T1.

## Failure modes (per new codepath)

| Codepath | Failure | Test? | Handling | User sees |
|---|---|---|---|---|
| renderVenues | corpus file **absent** (404, older trip) | add | `loadJson` fallback `[]` | section omitted, no throw |
| renderVenues | corpus file **present but unparseable** (hand-edit typo) | add | `loadJson` catch → `[]` **silently** ⚠️ | section vanishes; user thinks data lost — see below |
| renderVenues | malformed row (no name) | add | `isObject` filter + skip falsy (as renderFood) | row skipped, not a blank view |
| renderVenues | unsafe source_url | add | reuse `safeUrl` → `#` | inert link |
| collectVenues | candidate already promoted into a corpus (dupe) | add | dedupe by url-key/id; candidate in 待分類 only | shown once |
| placement-promote | `--to <unknown corpus>` | add | reject before write | error, files untouched |
| placement-promote | wrong-shape write (food fields into desserts) | add | per-corpus normalizer (T6) | clean corpus row |
| placement-promote | corpus file absent (older trip) | add | create-if-missing then append | promote succeeds |

**Critical-gap candidate (codex #9):** corpus-present-but-unparseable is silent
today (`loadJson:46-53` catches parse errors → fallback). With 4 more hand-editable
files this hides a whole section with no signal. **Fix (T-parse):** distinguish
"absent" (fine, omit) from "present but invalid" (render a subtle `⚠️ 甜點 載入失敗`
affordance, not silent). Promotes this from silent-and-unhandled to handled.
