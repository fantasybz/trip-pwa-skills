# ②-A.1 FSA write-back — manual smoke checklist (T8)

The mocked e2e (`templates/tests/edit-mode.spec.ts`) proves the WIRING. It cannot
prove the real File System Access semantics — transient user activation, permission
persistence across reload, structured-clone of the directory handle, real
`createWritable().close()`, and the SW cache refresh. Run this once on real Chrome
desktop before shipping v0.6.1.

## Setup
1. Scaffold (or reuse) a trip, then serve it locally:
   ```
   cd <trip-out-dir> && python3 -m http.server 8799 --bind 127.0.0.1
   ```
2. Open `http://localhost:8799/` in Chrome desktop (NOT Safari, NOT a remote host).

## Checks
- [ ] **Gating:** on `localhost`, enter 編輯模式 on 口袋名單 → the 「📁 連接 data/ 資料夾」button is visible. (On a `file://` open or a non-localhost host it must be ABSENT.)
- [ ] **Connect + health check:** click connect, pick the trip's `data/` folder, grant readwrite → chip shows 「已連接 · 自動寫回」. Picking the WRONG folder (e.g. Home) → 「這看起來不是這個行程的 data/ 資料夾」, no write.
- [ ] **Write-through:** paste a Reel / add a venue → within ~1s open `data/food.json` (or the routed corpus) on disk → the new entry is there, 2-space-indented, trailing newline (byte-equal to the CLI writer).
- [ ] **Survives IndexedDB clear:** DevTools → Application → IndexedDB → delete `trip-edit-overlay` → reload → the edit still shows (it was read from disk, not the overlay).
- [ ] **Undo restores on disk (Codex #1):** remove a venue (file updates without it) → tap 復原 → within ~1s the on-disk file is re-written WITH the venue back. (The bug this guards: an emptied delta used to skip the file, leaving the delete stale on disk.)
- [ ] **Promote ordering:** promote a 待分類 candidate → the target corpus file gains the venue; `feed_candidates.json` loses it. Both land; no orphan.
- [ ] **Permission re-grant after reload:** reload the tab → the chip may show 「已連接 · 待授權」 → the first edit prompts for permission once (a normal step), then writes resume.
- [ ] **Degrade on revoke:** revoke the site's file permission mid-session (site settings) → next edit shows 「寫回磁碟失敗,請先匯出」; the IndexedDB overlay + 匯出我的資料 still work.
- [ ] **No VersionError:** with edits flowing, confirm no `VersionError` in the console (fsa.js uses a separate `trip-fsa-handle` DB, not the overlay DB).

## Known limitation (do NOT test against)
Concurrent tab / CLI ingest / external editor touching the same `data/` between
page-load and a flush is silently clobbered by the whole-file write (single-writer
assumption). Staleness detection is deferred (T9).
