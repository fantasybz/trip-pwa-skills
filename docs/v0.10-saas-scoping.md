## The thesis tension (read this first)

The README sells this project as **the missing middle** between a chat concierge
and a hosted SaaS: *"a compiler… a static PWA you own, deploy to GitHub Pages…
not a chat answer, not a hosted SaaS."* The whole pitch is **you own it, you
don't rent it.**

A multi-tenant hosted SaaS — accounts, we host your trip, we hold your data — is
the exact thing the project defines itself against. Building it doesn't extend the
thesis; it abandons it. So "③ = SaaS" is the wrong default framing. The real
question is: **is there a hosted surface that helps without betraying ownership?**

## Three shapes for ③, honestly

### ③-A — hosted shared enrich-proxy (❌ not worth it)
A single hosted CORS proxy so non-coders can use an OpenAI key without deploying
their own Worker. **Rejected:** it reintroduces the exact footgun ②-B avoided —
users' BYOK keys transit *our* host, so we'd have to be trusted not to log them,
and a shared proxy is an abuse magnet (rate-limit, origin-allowlist headaches).
The v0.10 OpenAI proxy on-ramp (`docs/openai-proxy.md`) already solves the
non-coder gap *better*: Anthropic needs no proxy at all, and OpenAI users get a
copy-paste Worker they own + an OpenRouter no-Worker path. No shared key-handler
should exist.

### ③-B — hosted template gallery + one-click "deploy your own" (✅ the only thesis-safe one)
A static showcase site: a few real example trips (the Tokyo seed, the Hong Kong
dogfood), each with a **"Fork & deploy your own"** button (GitHub template repo →
Pages). We host *discovery and a deploy path*, never anyone's trip data. This
productizes the weakest part of the funnel (people don't know what the output
looks like or how to start) **without** holding data or keys. It's the natural
"③" that doesn't contradict ownership. Still: it's marketing/distribution, not a
product capability — only worth it once there's pull.

### ③-C — full multi-tenant SaaS (❌ betrays the thesis)
Accounts, hosted trips, billing, our DB. Needs auth, storage, a privacy posture,
payment, support — months of work — and it converts "you own it" into "you rent
it." Off the table unless the project's identity deliberately changes.

## Recommendation

**Don't build ③ now.** The honest next-best uses of effort, in order:

1. **Land ②-B properly** (this v0.10 PR): docs refreshed, OpenAI on-ramp,
   verifier-precision A/B staged. This is the real "finish what's shipped" work.
2. **Run the precision experiment** (`docs/verify-precision-experiment.md`) when a
   key's handy — quiet the ⚠️ warning if the A/B clears the gate.
3. **If — and only if — other parents ask to use it**, build ③-B (the gallery +
   fork-to-deploy). Until there's pull, a gallery is a site nobody visits.

### What would have to be true to start ③-B
- ≥ a handful of non-me users have actually deployed a trip from the skills, and
- the friction they hit is *discovery/starting*, not a missing capability, and
- it's **post-trip** (the 2026-07-25 Tokyo trip is the live priority until then).

None hold today.

## Reels ingest v2 — explicitly post-trip

The v1 ingest router (caption/URL → corpus, with `feed_candidates.json` as the
deferred-placement pool) is shipped and dogfooded. "v2" = smarter placement
(infer day/anchor/time from caption), batch dedup across a whole feed dump, and
auto-promote heuristics. It's **deferred to post-trip on purpose**: designing it
well needs real post-trip feed volume to test against, and it competes with
nothing the live trip needs. Revisit after 2026-07-31.
